Context
You have access to the upsaclass project in the GitHub folder (Next.js + MongoDB + Tailwind + shadcn/ui + Framer Motion + FullCalendar). The system already has:

Public homepage with FullCalendar (lecture schedule, search, filters)

Admin dashboard for buildings, halls, lecturers, courses, timetable slots

Authentication with roles: student, rep, admin

Now implement admin management of course reps and the complete rep request & approval workflow.

PART 1: Admin – Manage Course Representatives
1.1 Add a Course Rep
Admin navigates to a new section: "Course Representatives" (tab in admin dashboard).

Form to add a rep:

Field	Type	Notes
Student ID	Text	Required, becomes username for login
Full Name	Text	Required
Course	Dropdown	List of all courses – the rep will represent this course
Temporary Password	Text	Admin sets a password (will be hashed before saving)
On submit:

Create a new User with:

studentId = entered ID

passwordHash = bcrypt hash of the password

fullName = entered name

role = "rep"

courseRepFor = selected course's ObjectId

enrolledCourses = optionally auto-enroll the rep in that course? (Up to you, but rep should at least see that course's schedule)

Show success toast.

1.2 View All Reps
Table showing: Student ID, Full Name, Course Represented, Created At

Actions: Change Password, Delete

1.3 Change Rep Password
Button opens a dialog with:

New password field (with confirmation)

On submit: hash and update passwordHash in DB.

Show toast on success.

1.4 Delete Rep
Confirmation dialog (shadcn AlertDialog).

On confirm: delete the user document from users collection.

Show toast, refresh table.

PART 2: Course Representative – Login & Dashboard
2.1 Login
Rep uses Student ID and password set by admin.

On successful login, session stores role: "rep" and courseRepFor.

2.2 Rep Dashboard (after login)
Redirect to /rep/dashboard (or enhance student dashboard with rep-specific UI).

Dashboard shows:

Welcome message with rep's name.

Courses I Represent card/list: displays the course(s) they are rep for (one rep per course in this design, but could be multiple). Each course shows:

Course code & name

Next upcoming lecture (day, time, current hall)

Button: "Request Reassignment"

My Requests section/link to view all requests (status: pending/approved/rejected).

(Optional) Their personal timetable (if also enrolled in other courses).

PART 3: Request Reassignment – Full Flow (as you described)
3.1 Rep Clicks "Request Reassignment"
Opens a Dialog (shadcn/ui Dialog) with form.

3.2 Form Fields (with dynamic logic)
Field	Type	Behavior
Course	Dropdown	Pre-selected with the course they represent (or allows selection if multiple).
Date	Date picker	Only shows future dates when the course has a scheduled session. On date change, refresh available halls.
Start Time	Display only	Shows the scheduled start time for that course on that date.
Current Hall	Display only	Shows the currently allocated hall (could be default or an override).
Preferred Hall	Dropdown	Populated with available halls at that date/time – halls that are not double-booked and have capacity ≥ course enrollment.
Reason	Textarea	Required. Examples: "Double allocation", "Projector broken", "AC not working".
Submit	Button	Sends POST to /api/requests.
3.3 Submit Request
Request stored in reassignmentrequests collection with:

requestedBy: rep's user ID

slot: the timetable slot ID for that specific course session (based on date + day + time – you'll need to map date to a recurring slot)

preferredHall: selected hall ID

reason: text

status: "pending"

createdAt: now

After submit: close dialog, show success toast, redirect to "My Requests" page.

3.4 "My Requests" Page
List of all requests made by this rep.

Each request shows:

Course name

Date and time of session

Preferred hall

Status badge (pending = yellow, approved = green, rejected = red)

Admin comment (if rejected)

If pending: a Cancel button (DELETE /api/requests/{id}).

Use Framer Motion staggered list animation.

PART 4: Admin – Approve/Reject Request (with Hall Selection)
4.1 Admin Dashboard – Pending Requests Tab
List of all pending requests (cards or table).

Each request shows:

Rep name & student ID

Course name

Original hall (from timetable slot)

Preferred hall

Reason

Date/time of session

4.2 Approval Flow (as you described)
Admin clicks "Approve" on a request.

Opens a dialog:

Shows the requested session details.

Dropdown: "Select new hall" – populated with available halls at that date/time (excluding the original hall if it's free? Actually show all free halls).

Admin chooses a hall (could be the preferred one or another).

Optional: Admin comment.

On confirm:

Create a new Allocation record:

slot: the timetable slot ID

hall: selected hall ID

isActive: true

effectiveFrom: the date of the session (or immediate for recurring? For a single session, set effectiveFrom = session date, effectiveUntil = next day or null for permanent override? – decide based on your design: one‑time override or permanent until changed. I recommend one‑time override for a specific date.)

Update the request: status = "approved", approvedNewHall = selected hall, resolvedAt = now.

Generate notifications for all students enrolled in that course: Notification collection, each with user = student ID, message = "Your {courseCode} lecture on {date} at {time} has been moved to {hallName}.", isRead = false.

Show success toast.

4.3 Rejection Flow
Admin clicks "Reject", enters a comment.

Update request: status = "rejected", adminComment = comment.

Notify only the rep (create a notification for the rep).

PART 5: Student & Homepage Updates – Show Rescheduled Badge
5.1 Student Dashboard
When fetching timetable for a student, for each timetable slot, check if there is an active allocation for that specific date (if using date‑specific overrides) or for the current week.

If an override exists, display the new hall and show a badge: "Rescheduled" (or a 🔄 icon).

In the timetable grid, style the cell with a subtle red/orange border or a tooltip.

5.2 Homepage FullCalendar
The public calendar (homepage) should also reflect active allocations.

When building events for the calendar, for each recurring timetable slot, check if there is an allocation override for that specific date.

If yes, use the override hall in the event title/description and add a class fc-event-rescheduled to style differently (e.g., dashed border, red dot).

Event tooltip/popup should show: "Original hall: LT001 → Rescheduled to LT003".

PART 6: API Endpoints Summary
Method	Endpoint	Description	Access
POST	/api/admin/reps	Create a new rep (with student ID, name, course, password)	Admin
GET	/api/admin/reps	List all reps	Admin
PUT	/api/admin/reps/{id}/password	Change rep's password	Admin
DELETE	/api/admin/reps/{id}	Delete a rep	Admin
GET	/api/requests/my	Get logged-in rep's requests	Rep
POST	/api/requests	Submit a reassignment request	Rep
DELETE	/api/requests/{id}	Cancel pending request	Rep
GET	/api/halls/available?day=Mon&date=2025-04-15&time=10:00&courseId=...	Returns available halls (excluding conflicting, capacity sufficient)	Rep, Admin
PUT	/api/admin/requests/{id}/approve	Approve request (admin selects hall)	Admin
PUT	/api/admin/requests/{id}/reject	Reject request	Admin
GET	/api/timetable/events?date=2025-04-15	Returns events for a specific date (with override resolution)	Public
PART 7: Database Schema Additions (if not already present)
reassignmentrequests collection
json
{
  "_id": ObjectId,
  "requestedBy": ObjectId("user_id"),
  "slot": ObjectId("timetableslot_id"),
  "preferredHall": ObjectId("hall_id"),
  "reason": "string",
  "status": "pending|approved|rejected",
  "adminComment": "string",
  "approvedNewHall": ObjectId("hall_id"),
  "createdAt": Date,
  "resolvedAt": Date
}
allocations collection (for overrides)
json
{
  "_id": ObjectId,
  "slot": ObjectId("timetableslot_id"),
  "hall": ObjectId("hall_id"),
  "isActive": true,
  "effectiveFrom": Date,   // specific date for one‑time override
  "effectiveUntil": Date | null
}
notifications collection
json
{
  "_id": ObjectId,
  "user": ObjectId("user_id"),
  "message": "string",
  "isRead": false,
  "relatedRequest": ObjectId("request_id"),
  "createdAt": Date
}
PART 8: Acceptance Criteria
Admin can create a rep (student ID + password), change password, delete rep.

Rep logs in with student ID + password and sees dashboard with "Courses I Represent".

Rep can request reassignment for a future lecture, with available halls dynamically filtered.

Rep sees request status in "My Requests" and can cancel pending requests.

Admin sees pending requests with badge, can approve (choose any free hall) or reject.

On approval, an allocation override is created and notifications are sent to all enrolled students.

Student dashboard and homepage calendar show the updated hall with a "Rescheduled" badge/indicator.

All UI elements use shadcn/ui components, Tailwind CSS, and Framer Motion animations.

Mobile responsive.
