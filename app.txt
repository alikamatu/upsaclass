RESEARCH DOCUMENT: DESIGN AND IMPLEMENTATION OF A WEB-BASED CLASSROOM ALLOCATION AND RESCHEDULING SYSTEM
1. INTRODUCTION
1.1 Background
Tertiary institutions in Ghana, such as the University of Professional Studies, Accra (UPSA), typically operate with fixed timetables for lecture hall allocation each semester. While this approach works under normal circumstances, it fails dynamically when disruptions occur – faulty equipment, double allocations, or last-minute venue changes. Currently, course representatives and lecturers manually search for available halls, leading to wasted time, missed lectures, frustration, and disorganized rescheduling.

1.2 Problem Statement
Most tertiary institutions rely on fixed timetables. When lecture halls become unavailable due to faulty learning equipment, double allocation, or unforeseen issues, no centralized system exists to:

Quickly find an available hall

Request a reassignment

Notify affected students

Track changes over time

This results in inefficiency and academic disruption.

2. PROJECT OBJECTIVES
2.1 General Objective
To design and implement a web-based classroom allocation and rescheduling system that improves access to lecture hall information and minimizes disruptions caused by classroom unavailability.

2.2 Specific Objectives
#	Objective
1	Design a centralized database for managing classrooms, courses, and lecture schedules.
2	Develop a web-based platform that displays allocated lecture halls for courses.
3	Enable students to search and view their assigned lecture halls and updated class locations.
4	Implement role-based access control (RBAC) for students, course representatives, and administrators.
5	Allow authorized users (course reps) to request classroom reassignment during disruptions.
6	Enable students to view their personalized class timetables.
3. PROJECT SCOPE
3.1 In-Scope Features
Web-based access via browser (responsive design)

Student course search and lecture hall viewing

Viewing of rescheduled or updated classroom information

Classroom availability checking (real-time)

Classroom reassignment request functionality (by course reps)

Administrator approval and management dashboard

Personalized class schedule access per student

In-app notifications (bell icon, unread badges)

3.2 Out of Scope
Mobile application (progressive web app may be added later)

Classroom equipment status tracking (e.g., broken projector flag)

SMS or email notifications (only in-app)

Full university-wide timetable automation (manual entry of master schedule)

Automated maintenance of halls

Integration with existing institutional or academic management systems (e.g., legacy ERP)

4. SYSTEM FEATURES (BY USER ROLE)
4.1 Student
Secure login with student ID and password

View personalized weekly timetable (course, time, hall, lecturer)

Search for any course to see its current hall and schedule

Receive in-app notifications when a hall changes for their enrolled courses

View status of requests (if they are also a rep – see below)

4.2 Course Representative (a Student with elevated privileges)
All student features

Submit a classroom reassignment request for a specific course session

Provide reason for request (e.g., double allocation, broken AC/projector)

Suggest a preferred alternative hall (system checks availability)

Track request status: pending → approved/rejected

4.3 Administrator
Full CRUD operations: lecture halls, courses, timetable slots

View all current allocations (master + overridden)

Approve or reject reassignment requests

When approving, select an available hall (system prevents conflicts)

Publish changes → instantly updates student views + triggers notifications

View system audit logs (who requested what, when, resolved by whom)

5. SYSTEM ARCHITECTURE & TECHNOLOGY STACK
Layer	Technology	Purpose
Frontend	Next.js 14 (App Router)	React framework with server components & routing
Styling	Tailwind CSS + Framer Motion	Utility-first CSS + smooth animations
Backend	Next.js API Routes (Node.js)	Serverless functions for business logic
Database	MongoDB (with Mongoose ODM)	NoSQL, flexible schema for timetable data
Authentication	NextAuth.js (Credentials provider)	Session-based auth with role embedding
Hosting	Vercel / Local server	Easy deployment for Next.js apps
5.1 Why Next.js + MongoDB?
Next.js provides both frontend and API routes in one project, simplifying deployment.

MongoDB handles semi-structured data like timetable slots, allocations, and requests without rigid joins.

Tailwind + Framer Motion allows rapid UI development with smooth page transitions and feedback animations.

8. API ENDPOINTS (Next.js API Routes)
Method	Endpoint	Description	Access

10. SECURITY & ROLE-BASED ACCESS
Authentication: NextAuth.js stores session in HTTP-only cookie. Passwords hashed with bcrypt.

Authorization: Middleware checks token.role before allowing access to /admin/*, /rep/*.

API protection: Each API route validates session and role before processing.

Input validation: Mongoose schemas enforce data types; API routes sanitize strings.

MongoDB injection: Avoided by using Mongoose parameterization.
